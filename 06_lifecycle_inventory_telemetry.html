Coordination model: In this RACI, “Cequence R” means the Cequence SaaS vendor (Cequence Ops) executes the action; SecEng coordinates, reviews evidence, and maintains the operating model. SecEng does not generate keys or deploy certificates directly.

Approval routing requirement: Cequence/OKAPI teams may generate keys, create CSRs, and submit certificate requests; however, issuance does not auto-complete. Every request must enter the DCERTS workflow and remain pending until PKI/DCERTS approves and issues the certificate. This applies equally to manual portal, DCERTS REST automation, and any future ACME/cert-manager path. For emergency revocation or emergency re-issue, PKI/DCERTS remains R/A and must be engaged as part of the incident runbook.

10.5 Certificate Inventory Schema

The certificate inventory is the target-state authoritative source of truth for renewal planning, incident response, and automation. A spreadsheet or Confluence table may serve as a bootstrap implementation. It remains non-authoritative until integrity controls are in place: a named data owner, restricted write access, immutable change history, validation rules, and a defined reconciliation process with DCERTS and endpoint scan results. Any roadmap item that depends on an “authoritative inventory” applies post-cutover. [tab:inventory-schema] defines the complete field set.

10.5.1 Inventory enrichment

Every workflow that creates or updates an inventory record MUST populate all 34 fields in [tab:inventory-schema]. DCERTS expiry alerts provide partial data only. Automation must enrich from: (a) certificate parsing (leaf+chain), (b) ownership/deployment lookup tables (CMDB/Confluence), and (c) change-management metadata. The detailed field-by-field sourcing model is maintained in the operational specification on Confluence.

10.5.2 Pattern: mTLS client certificate tracking (certificate_role=client_mtls)

Where inbound mTLS is enabled (consumer \(\rightarrow\) defender), certificate operations must track client certificates as first-class lifecycle objects rather than assuming a server-only model. Expired or revoked client certificates manifest as handshake failures and must be tracked for incident response. Onboarding patterns that rely on explicit customer identity binding also depend on this tracking.

Minimum inventory requirements (in addition to [tab:inventory-schema]):

  • Record each client certificate (or client certificate class) using certificate_role=client_mtls and the relevant endpoint_fqdn it authenticates to.

  • Populate issuer_ca and valid_to so client-cert expiry can be monitored and correlated to handshake failures.

  • Populate key_custody only where Cequence/OKAPI actually hold the client private key (for most external API consumers this will be external and therefore not applicable; do not invent custody).

Trust anchor tracking (defender-side verification):

  • Track the client-auth trust anchor (root/intermediate set) used by the defender as ca_bundle role and reference it in deployment_targets (defender pools).

  • Treat changes to this trust bundle as a high-risk change (it can break all client auth). Apply the same deployment safety gate as server cert changes, including a “known-good vs known-bad” mTLS probe (see Table 18).

10.5.3 Inventory schema enforcement

The inventory schema MUST enforce the relational model in 15: every certificate record anchors to exactly one endpoint, every certificate record has one or more deployment targets, and change records reference certificate records via an explicit junction (not flattened fields). Workflow rules block change closure until every linked certificate record has at least one deployment target and a populated rollback_ref. The detailed SharePoint List design (five lists: Endpoints, CertificateRecords, DeploymentTargets, ChangeRecords, ChangeRecordCertificateLink) is maintained in the operational specification.

The operating model recommends non-exportable key custody (HSM/KMS/Vault) as the target state for production private keys. The schema still allows key_custody=file_based so teams can truthfully capture legacy storage patterns; any such record should be treated as a remediation trigger with a named owner, a migration plan, and a time-bound risk acceptance.

10.6 Telemetry and Alerting Hooks (Early Detection)

Expiry, chain incompatibility, truststore drift, and mis-issuance require continuous monitoring to emit machine-detectable signals. Certificate lifecycle telemetry is a first-class architectural concern.

Telemetry design goals:

  • Alerts must trigger while there is still time to plan a change (weeks or days), not at the moment of failure.

  • Cequence (edge) and OKAPI (origin) are monitored independently; split custody must not create split blindness.

  • Every alert must include endpoint_fqdn and map to the inventory record_id and owner_team.

  • Where possible, prefer external (synthetic) verification of the served certificate over control-plane claims.

These signals can be implemented using the validation scripts in Appendix 20 as probes (e.g., a scheduled openssl s_client check) and exported into the monitoring stack (Datadog/Prometheus) with a paging integration (PagerDuty). Alerts must reference inventory identifiers and have an on-call owner.

The monitoring pipeline follows five stages: Observe (probes, DCERTS, defenders, inventory) \(\to\) Ingest (OneSplunk) \(\to\) Correlate by \(\mathcal{C}\) (endpoint_fqdn, environment, certificate_role) \(\to\) Decide (severity routing: Sev 1/2 \(\to\) PagerDuty, Sev 3+ \(\to\) Jira) \(\to\) Notify and feed into the M1 Triage Machine.

10.6.1 Splunk Integration (Cequence Events)

Cequence exports security events to OneSplunk via HEC. Index naming convention, HEC token configuration, and transport architecture are documented in the Splunk integration operational specification on Confluence.

Warning: HEC provides no event replay capability; events lost during Splunk outages cannot be recovered. Cequence UI retains up to 7 days of event history as a manual fallback.
