16 Recommendations

16.1 R1: Resolve DCERTS ACME Blocker

Recommendation. Engage AppViewX vendor to resolve the HTTP 500 (AVX_CERT-ACME-008) error blocking automated certificate issuance.

Conservative option:

  • Escalate to vendor with full debug logs and transaction traces

  • Establish SLA for vendor response (target: 5 business days)

  • Document manual fallback procedure for production use

Pragmatic option:

  • Continue manual DCERTS portal workflow while awaiting fix, but make it deterministic: standardised request templates, mandatory lint gates, and an explicit evidence pack per issuance/renewal.

  • If browser automation (Selenium/Playwright) is used, constrain it to read-only evidence capture and explicitly prevent it becoming an unofficial issuance control path. Minimum controls:

    • Read-only scope: automation may only export reports / capture page state; it must not submit forms, click “approve/issue/renew”, upload CSRs, or trigger any workflow transitions.

    • Separated identity: use a dedicated evidence-only service account with read-only entitlements; do not reuse any account that can request/approve/issue/renew.

    • Hard technical guardrails: enforce network allowlists and application-layer blocks so the automation cannot reach DCERTS endpoints associated with issuance actions (even if code changes under pressure).

    • Deterministic execution and auditability: full request/response logging, retry/backoff with alerting, and tamper-evident storage (hash + timestamp) for exported artefacts/screenshots.

    • Operational policy: any attempt to extend automation beyond evidence capture is treated as a change to a control and requires explicit security/PQI governance approval (tracked as a change record with a risk assessment).

  • Prefer an API-first interim automation path (DCERTS REST API direct integration) while ACME remains blocked; re-evaluate STAC (AWS-native certificate management) only if it can preserve split custody, auditability, and CA policy equivalence.

16.2 R2: Implement Certificate Inventory

Recommendation. Deploy a machine-readable certificate inventory and manage the transition from bootstrap (non-authoritative) to authoritative source of truth once integrity controls are operating.

Conservative option:

  • CMDB integration with custom certificate asset class

  • Automated discovery via certificate transparency logs and endpoint scanning

  • API-first design for automation integration

Pragmatic option:

  • Use the existing Confluence certificate inventory page as the bootstrap inventory (not authoritative unless integrity controls are in place) using the defined schema

  • Put minimum integrity controls in place: named data owner, write access restricted by group, change history enabled, and validation rules for CN/SAN, dates, and environment

  • Reconcile weekly against DCERTS exports and endpoint scan results; treat reconciliations as evidence artefacts

  • Define the cutover criteria for “authoritative” status (e.g., automated ingest/sync + validation + workflow) and a migration path to a machine-readable system with API access

16.3 R3: Deploy Control Gates

Recommendation. Implement deterministic validation gates for CSR, certificate, and deployment stages.

Conservative option:

  • Integrate ZLint or certlint for CSR and certificate validation

  • Automated chain validation with OpenSSL and JVM compatibility testing

  • Gate failures block pipeline with mandatory remediation

Pragmatic option:

  • Start with manual checklist validation

  • Add shell script wrappers for OpenSSL validation commands

  • Capture validation output in evidence pack

16.4 R4: Establish Emergency Templates

Recommendation. Pre-approve emergency change templates with defined escalation paths.

Conservative option:

  • Define playbooks for: leaf revocation, key compromise, intermediate CA revocation

  • Pre-approved change templates with 4-hour SLO

  • Quarterly tabletop exercises

Pragmatic option:

  • Document emergency contact matrix (L4 on-call, PKI team, Cequence support)

  • Add revocation scenarios to existing renewal planning meetings

  • Test rollback procedure annually

16.5 R5: Auto-Generate Change Review Packs

Recommendation. Automate generation of the Change Review Pack (CRP) for certificate renewal changes. The current process is fully manual: each renewal requires a CRP populated from scratch with certificate details, stakeholder contacts, PVT criteria, and backout procedures. At scale this becomes a bottleneck (analogous to the 2–3 hour CIP generation problem observed by other platform teams). Auto-populating the CRP from the certificate inventory eliminates transcription errors and reduces SecEng coordination time from hours to minutes per renewal (see Table 2).

Conservative option:

  • Build a CRP generator that pulls certificate metadata (CN, SAN, issuer, expiry, custody zone, deployment targets) from the inventory and populates the standard CRP template fields automatically

  • Include pre-filled PVT criteria (fingerprint check, chain validation, SLO thresholds) drawn from the G3 gate specification

  • Generate the backout procedure from the rollback_ref in the inventory (previous certificate version and deployment state)

Pragmatic option:

  • Create a parameterised CRP template (Confluence or SharePoint) that pre-populates from inventory export

  • Require only manual review and approval signature rather than manual data entry

  • Target: CRP generation time from hours to minutes per renewal

16.6 R6: Confirm Secrets Management Ownership

Recommendation. Confirm that the Cequence SaaS vendor and OKAPI platform team maintain secrets custody (private keys, EAB keys, TSIG secrets) via their respective control planes (AWS Secrets Manager, Vault/ESM). Document the custody boundary explicitly so that audit and incident response know which team to engage for key-related operations.

Conservative option:

  • Obtain written confirmation of custody ownership per environment (prod, non-prod) from Cequence vendor and OKAPI Ops

  • Ensure the custody model is captured in the certificate inventory (key_custody field)

  • Verify that vendor SOC2 or equivalent attestation covers key management controls

Pragmatic option:

  • Document current custody arrangements in the operational specification

  • Add custody verification to the annual review cadence

16.7 R7: Deploy ORR Gate and Trust Anchor Allowlist

Recommendation. Deploy the Operational Readiness Review gate (Section 4.5.3) as a blocking deployment control and operationalise the trust anchor allowlist (Section 7.2) for non-Telstra CA onboarding.

Conservative option:

  • Integrate KUDOS Compare API into the renewal workflow; enforce ORR form submission before any certificate reaches production

  • Complete CA due diligence (Table 9) for Amazon Trust Services (ACM) as the first non-Telstra CA entry

  • Validate ACM delegation on a single non-production defender endpoint (Section 7.1.1, fastest disproof test)

Pragmatic option:

  • Run ORR in advisory mode (log risk scores, do not block) for one renewal cycle to baseline risk score distribution

  • Add Amazon Trust Services to the allowlist with Tier 2 controls; defer Tier 3 (BYO domain, external CA) until Phase 2 demand materialises

  • Use CloudWatch/SNS event forwarding as the compensating control for I5 (approval boundary) on ACM-managed certificates

16.8 R8: Enable Domain Authorisation for Custom Domains

Recommendation. Operationalise domain authorisation (Section 7.1) for custom domain routing on Cequence defenders.

Conservative option:

  • Enforce domain allowlist in the CSR lint gate (Section 11.1); reject any SAN not on the list

  • Deploy NGINX location block routing incrementally per-API (Section 7.1, custom domain routing approach)

  • Require domain authorisation evidence (Table 8) attached to every certificate object before issuance

Pragmatic option:

  • Start with Phase 1 (Telstra-owned domains only); document the Phase 2 gate criteria but do not enable BYO domain onboarding

  • Pilot the NGINX location block routing on one custom domain in non-production before applying to production defenders

  • Track domain authorisation expiry (12-month validity) in the certificate inventory with automated renewal alerts

  • For Phase 2 BYO domains: monitor domain_registration_expiry against certificate not_after; flag for immediate review if the domain registration expires before the certificate
