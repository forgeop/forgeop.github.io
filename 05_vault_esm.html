9 HashiCorp Vault Integration (ESM)

The Enterprise Secrets Management (ESM) platform based on HashiCorp Vault provides the target-state secrets management solution. Vault/ESM is the single secrets control plane for governance (policy, access control, rotation, audit). Kubernetes Secret objects are a workload-facing delivery format only (a distribution cache) and must not be treated as the governance boundary or system of record for private keys.

Some workflows may still materialise secrets into Kubernetes (e.g., operator patterns, ingress controller TLS termination, cert-manager outputs). These must be constrained so they do not become a second control plane:

  • The source of truth for sensitive credentials (EAB keys, TSIG secrets, API tokens) must be Vault/ESM (or, until Vault is available, AWS Secrets Manager as an interim control plane), with Kubernetes Secrets populated by a controlled sync mechanism.

  • Any Kubernetes Secret that contains a private key is treated as runtime only: namespace-scoped, least-privilege RBAC, encryption-at-rest enabled for etcd, and a defined rotation/expiry process.

  • Teams must not create parallel “manual” stores (SharePoint, ad-hoc copies) for private keys or perpetual credentials; any exception must be time-bound and recorded as a risk acceptance.

9.1 ESM Architecture Overview

  • Phase 1 (Current): Vault cluster on AMP Kubernetes (AWS EKS) in an Australian region / AU-resident landing zone (requires confirmation; no evidence cited; see minimum evidence checklist below).

  • Phase 2 (Planned): Vault cluster in Azure tenancy in an Australian region (requires confirmation).

  • Phase 3 (Planned): On-premises deployment on PUMA in Australia (requires confirmation).

Multi-cloud sequencing is acceptable only if every phase used for production key operations is demonstrably AU-resident. Any phase that is not AU-resident MUST NOT be used for production private key generation, signing, or storage. It may only store non-sensitive artefacts (e.g., public certificates/CA bundles) until an Australian-hosted deployment is available and approved.

Minimum evidence required per phase (before production use):

  • Cloud region/site and tenancy/landing-zone identifier (explicitly AU)

  • Confirmation that backups, replication, and log sinks remain in Australia (including DR patterns)

  • Key handling model (HSM/KMS/Vault), including whether keys are exportable

  • Migration approach: no cross-cloud private key movement (rotate by generating a new key in the target phase and re-issuing via DCERTS)

9.2 Certificate Lifecycle Capabilities

HashiCorp Vault provides native PKI capabilities relevant to certificate lifecycle:

HashiCorp Vault PKI capabilities from ESM BRD.
Capability Description
PKI secrets engine Generate private keys and CSRs; can operate as a CA/issuer only if explicitly delegated and governed. In this operating model, DCERTS remains the enterprise issuance portal/authority for production leaf certificates.
Certificate signing Support patterns where Vault participates in an issuance workflow (e.g., key+CSR generation, policy enforcement, storage, and distribution) while an external enterprise CA/portal (DCERTS) performs the signing and maintains issuance records.
Perpetual secrets Create non-expiring credentials (for example, EAB keys / TSIG secrets). Certificates must not be perpetual: the target-state is short-lived certificates (e.g., 90-day) with automated renewal (ACME or DCERTS REST/operator); until that automation is available, the operating model assumes DCERTS-issued certificates up to the 398-day maximum with deterministic, early, manual renewals (see Section 13).
Encryption service All secrets encrypted at rest (AES-256-GCM) and in transit (TLS 1.2+)
Fine-grained policy ABAC policies attached to secrets or secret hierarchies
HSM integration Master key storage in external HSM module
Agent injector Runtime secrets injection for Kubernetes pods (sidecar pattern)

9.3 Integration Pattern

Vault integration does not imply centralised certificate issuance or key custody. If adopted, Vault must operate as a multi-tenant secrets platform with strict isolation between Cequence and OKAPI custody domains (separate namespaces/mount paths, separate auth roles, separate audit scopes, and no cross-team read access to private key material).

Governance boundary:

  • Issuer of record: For production leaf certificates, DCERTS is the issuer of record and the authoritative lifecycle system (request/approval/issuance/renewal/revocation and audit trail).

  • Vault role in this model: Vault handles secrets custody, access control, distribution, and (optionally) CSR/key generation on behalf of workloads. It is not an enterprise issuing CA for externally trusted certificates unless PKI governance explicitly delegates that authority.

  • What “external CA root” means here: It means the enterprise/public trust chain that workloads must trust/serve (e.g., DCERTS-selected CA chain). It does not mean Vault is permitted to mint certificates from a DCERTS root.

  • Key custody: Private keys for Cequence edge certificates remain under Cequence-approved custody (HSM/KMS/Vault as approved); issuance authority remains with DCERTS.

Example A (only if governance explicitly delegates issuing authority to Vault):

# Enable PKI secrets engine
vault secrets enable -path=pki pki

# NOTE: This is an internal-PKI pattern.
# You cannot "import" a public/enterprise CA (e.g., DCERTS-selected DigiCert chain) into Vault and
# have Vault issue from that external root. If Vault is delegated issuing authority, it must operate
# as its own CA with an explicitly governed trust model.

# Configure Vault-managed CA (example)
vault write pki/root/generate/internal common_name="Example Internal Root" ttl="87600h"

# Create role for certificate issuance
vault write pki/roles/cequence-defender \
    allowed_domains="api.telstra.com,okapi.telstra.com" \
    allow_subdomains=true \
    max_ttl="2160h" \
    key_type="rsa" \
    key_bits=2048 \
    require_cn=true

Example B (governance-safe default): Vault as secrets custody/distribution around DCERTS-issued leaf certificates:

# Enable KV secrets engine (store EAB/TSIG secrets, issued certs, and keys)
vault secrets enable -path=kv kv-v2

# Store CA chain / trust bundle used by workloads for validation (example)
vault kv put kv/pki/trust/dcerts ca_bundle=@dcerts-chain.pem

# Store issued leaf + key after DCERTS issuance (example; key generation/custody must be approved)
vault kv put kv/pki/cequence/defender/api.telstra.com \
  tls_crt=@leaf.pem tls_key=@leaf-key.pem

# DCERTS remains the issuing system; any automation should submit CSRs to DCERTS
# via approved interfaces (portal workflow, REST API, or ACME once unblocked).
