7 Non-Telstra CA Support, Custom Domains and Governance

Non-Telstra CA onboarding and custom domain enablement require three controls: domain authorisation (phased lifecycle), trust anchor allowlisting (CA due diligence and trust profile bindings), and a governance framework (tiering, ownership, review cadence, decommission triggers). The governance framework prevents entropy as the estate grows.

Both are dimensions of the certificate lifecycle in Section 4, not separate lifecycles. Every certificate object \(\mathcal{C}\) has a domain authorisation status and a trust profile binding. The same state machine (Table 5) applies, with additional gates per dimension.

7.1 Domain Authorisation

“Support custom domains” is a lifecycle and controls requirement, not a promise that any arbitrary domain is accepted. Domain scope is phased (Section 1): Phase 1 covers Telstra-owned domains; Phase 2 (future, gated) covers BYO domains.

Domain authorisation evidence requirements.
Phase Required evidence Verification method
Phase 1 DNS zone ownership attestation from Telstra DNS team, or delegated zone evidence with zone owner approval Manual verification against DNS team records or ServiceNow/CMDB delegation records
Phase 2 All of Phase 1, plus: domain ownership proof via DNS TXT record, validation method approval (DNS-01/HTTP-01), signed operational support boundary acknowledgement Automated DNS lookup for TXT record; manual review of validation method and support acknowledgement

Gate binding (enforces Invariant I7): The CSR lint gate (Section 11.1) rejects any SAN/FQDN not present in the domain allowlist. The deployment safety gate (Table 18) requires domain authorisation evidence for any new or changed SAN. Authorisations expire after 12 months; failure to re-validate at renewal triggers decommission.

Domain uniqueness constraint: Each endpoint_fqdn maps to exactly one tenant (Cequence Ops or a specific OKAPI customer). The CSR lint gate enforces this: a SAN that already exists in the inventory under a different owner_team is rejected with a conflict error referencing the existing record. Conflict resolution requires Security Architecture approval (RACI “A” in Table 11). This constraint prevents cross-tenant certificate issuance for the same domain and eliminates the routing ambiguity that would result from duplicate SAN bindings on shared defenders.

Custom domain routing (implementation approach): Where custom domains front-end both Apigee (legacy) and OGW (target), traffic routing uses NGINX location block base-path matching on the Cequence defenders (not regex). Each API migrating from Apigee to OGW receives an explicit location ^~ directive on the shared virtual host (e.g. management.api.telstra.com, products.api.telstra.com). The default location / routes to Apigee; specific paths override to OGW. Deployment is incremental per-API and per-environment, with rollback by removing the override block.

7.1.1 ACM delegation model for vendor-managed certificates

Vendors (e.g. Cequence) may manage their own certificate renewals via AWS Certificate Manager (ACM) under specific conditions.

  • Hard constraint: Split custody must hold. Private keys generated within the vendor’s custody zone must not cross boundaries.

  • ACM properties: ACM generates, stores, and rotates private keys within AWS-managed HSMs. Keys are non-exportable. ACM-issued certificates are domain-validated (DV) via DNS or email. Renewal is fully automated for ACM-issued certs on integrated AWS services (ELB, CloudFront, API Gateway).

  • Alignment with enterprise standards: ACM delegation is an industry-standard pattern for SaaS platforms on AWS. AWS holds SOC 2 Type II, ISO 27001, and PCI DSS certifications for ACM. The AWS Shared Responsibility Model places key custody with AWS. This is acceptable when the vendor’s deployment runs entirely within AWS and keys are not shared.

ACM delegation is acceptable for the Cequence custody zone on Telstra-owned domains. ACM-issued certificates use a non-DCERTS CA (Amazon Trust Services), placing them at Tier 2 minimum under the tiering model (Table 10). Conditions:

  1. The certificate’s server_trust_profile_id references a profile where ACM (Amazon Trust Services) is an approved issuer on the trust anchor allowlist.

  2. Domain validation evidence (DNS TXT record or CNAME) is captured in the evidence ledger.

  3. Renewal events are forwarded to the CERTLIFECYCLE Jira ticket (via CloudWatch Events or SNS) for auditability. The vendor manages the renewal; SecEng retains visibility.

  4. The OKAPI custody zone retains independent certificate management (ACM delegation does not extend across custody boundaries).

The main counterargument: ACM delegation removes SecEng from the renewal approval loop entirely. Certificates are issued by Amazon Trust Services (DV only), not by the enterprise CA behind DCERTS, so I5 (approval boundary) does not apply in its current form. I5 governs DCERTS-issued certificates specifically. ACM-managed certificates use a distinct trust model: the cloud provider is both issuer and key custodian. Compensating controls include (a) a trust anchor allowlist entry for Amazon Trust Services, (b) continuous SNI probes detecting unexpected certificate changes, and (c) CloudWatch/SNS event forwarding for SecEng visibility into every renewal. If these controls prove insufficient, restrict ACM delegation to non-production environments only.

To validate, deploy a single ACM-managed certificate on a non-production defender endpoint and verify: (a) renewal occurs without manual intervention, (b) the CloudWatch event fires and is captured in the Jira ticket, and (c) the SNI probe confirms the new fingerprint within the SLO window. If any check fails, the delegation model requires additional compensating controls.

7.2 Trust Anchor Allowlist

The CA posture (Section 1) enforces an explicit allowlist of approved CA profiles. Case-by-case approval without allowlist entry is disallowed. Non-Telstra (external) CAs are permitted only after passing due diligence.

CA/trust profile due diligence checklist.
Check Evidence required Renewal
WebTrust / ETSI audit status Current audit report or public attestation URL Annual
Revocation capability CRL/OCSP responder URL; tested responsiveness Annual
Certificate policy (CP) review CP document reviewed for alignment with Telstra requirements On change
Chain compatibility Test chain validated against target truststores (OpenSSL 3.x, JVM, iOS/Android) On CA change
Operational support Named contact for incident escalation; SLA for revocation requests Annual

A trust profile (server_trust_profile_id) encapsulates: issuer constraints (allowed root/intermediate subjects or SKI fingerprints), certificate constraints (validity limits, key requirements, EKUs), rotation expectations, and expiry/sunset date. For mTLS, a client trust bundle (client_trust_bundle_id) defines the trusted CA set for client auth validation, required EKUs, and bundle version.

mTLS mode: Each certificate object records mtls_mode: disabled (server-auth only), optional (client cert requested but not required), or required (connection rejected without valid client cert). This determines whether client_trust_bundle_id is mandatory.

7.3 Governance Framework

Governance prevents entropy as the estate grows. Controls are enforced through tiering, ownership boundaries, decommission triggers, and review cadences.

7.3.1 Tiering

Certificate estate tiering model.
Tier Eligibility Controls
Tier 1 Telstra domain (Phase 1); DCERTS-issued; no mTLS Standard gates; annual review
Tier 2 Telstra domain; non-default CA profile or mTLS Enhanced due diligence; 6-month review
Tier 3 BYO domain (Phase 2) or external CA or custom assertion Full due diligence; quarterly review; time-bounded approval

Tier is computed from domain_phase, server_trust_profile_id, mtls_mode, and assertion_contract_version. Quotas apply: Tier 3 certificate count is capped (initial: TBD); trust profile count is capped (initial: 10 server, 5 client bundles); BYO domain onboarding is gated (initial: 0, Phase 2 not enabled).

7.3.3 Decommission triggers

Decommission triggers and grace periods.
Trigger Condition Grace
Domain auth expiry auth_valid_until reached without re-validation 30 days
Trust profile sunset Profile sunset date reached 90 days
Exception expiry Time-bounded exception not renewed Immediate
Owner unassigned owner_team invalid > 14 days 14 days

Domain registration lapse domain_registration_expiry earlier than certificate not_after (Phase 2 only) Immediate

When triggered: certificate flagged “decommission pending”; renewal blocked; escalation to owner and Security Architecture if certificate still valid.

7.3.4 Review cadence

Tier 1: annual. Tier 2: 6-monthly (includes trust profile validity, mTLS bundle currency). Tier 3: quarterly (includes exception status, risk acceptance currency). Each review produces an evidence artefact attached to the inventory record.

7.3.5 Exceptions

Exceptions must be: explicitly documented (justification, risk assessment, compensating controls); time-bounded (max 90 days Tier 2, 30 days Tier 3); reviewed at expiry (renew or remediate); owner-acknowledged. Expired exceptions without remediation block renewal until resolved.
