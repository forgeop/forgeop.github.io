Certificate Lifecycle Operating Model

Security Architecture and Engineering
L4 SecEng Team
Telstra Group

29 January 2026 Version 2.0

Abstract

This paper presents a certificate lifecycle operating model for Cequence API defenders in a split custody TLS environment. Five outage-driving failure modes (expiry, chain incompatibility, truststore drift, mis-issuance, key compromise) are analysed and translated into a three-gate control model with evidence requirements and repeatable runbooks. The model introduces an Operational Readiness Review (ORR) gate that uses risk scoring to govern deployment decisions. Technical artefacts include an inventory schema, CSR and certificate validation specifications, a deployment safety gate, and target-state Kubernetes manifests. ACME content is target-state reference only; all recommendations are implementable under current constraints (manual DCERTS workflows and REST API patterns) and designed to evolve toward higher automation without weakening custody boundaries.

1 Method, Scope and Source Artefacts

1.1 Reader model and viewpoint

The Cequence platform owner is accountable for availability, security, and auditability of customer-facing REST APIs protected by Cequence defenders. This operating model targets outage-driving certificate failures. It preserves split custody boundaries and mandatory PKI/DCERTS approvals.

1.2 Scope boundaries

  • In scope: REST API endpoints protected by Cequence defenders, including server certificates for inbound TLS termination and downstream TLS legs where Cequence is accountable for lifecycle coordination.

  • Out of scope: SOAP, SIP, SS7, Diameter, IMS signalling, and non-HTTP API protocols.

  • Tooling: Jira is the internal system of record for triage. Zendesk is a manual vendor-facing handoff until a vendor-approved integration path exists.

  • Cloud: The operating model must function in a cloud environment with explicit data residency constraints for private key operations.

1.2.1 Architectural boundary (TLS termination)

TLS terminates at the Cequence defender. The downstream OKAPI Gateway (OGW) receives traffic over a separate TLS session and cannot observe the client-facing handshake. This is a hard constraint: OGW cannot independently verify TLS version, cipher suite, or client certificate details and must rely on assertions from Cequence.

1.2.2 Domain scope (phased enablement)

The operating model phases domain (FQDN/SAN) support:

  • Phase 1 (current): Telstra-owned domains only. Domain must be in a Telstra DNS zone or an explicitly approved partner zone; authorisation evidence is ownership attestation from Telstra DNS team or delegated zone evidence.

  • Phase 2 (future, gated): Customer BYO domains. Requires additional gates: domain ownership proof (DNS TXT record), validation method approval, explicit offboarding path, and operational support boundary acknowledgement. Phase 2 requires an explicit architecture review and a risk acceptance decision.

The phased model ships safely with Telstra domains first, preserves extensibility for BYO domains, and prevents governance entropy.

1.2.3 CA posture (allowlist model)

The operating model enforces an allowlist of approved CA profiles (Decision A). Case-by-case approval without an allowlist entry is disallowed. Every certificate must reference a server_trust_profile_id from the trust anchor allowlist. Non-Telstra CAs are permitted only after passing due diligence and being added to the allowlist.

1.3 Method

  • Start from outage-driving failure modes, then design enforceable control gates and evidence requirements.

  • Treat runtime truth as authoritative. Deployed means served on the wire and proven by SNI probe and fingerprint.

  • Maintain custody boundaries. Inventory and workflows handle metadata only.

  • Use internal artefacts as primary inputs. Public standards are used as design guidance, not as a compliance claim.

3 Executive Summary

Cequence defenders terminate TLS (or mTLS) for inbound API traffic and establish a separate TLS session to upstream platforms. Two distinct certificates exist for the same customer-facing domain, each with independent private keys and different operational owners (Cequence Ops vs OKAPI Ops). This split custody creates a certificate lifecycle problem that is both technical and organisational.

Certificate issuance currently relies on manual DCERTS workflows and the DCERTS REST API. The target state is ACME automation (RFC 8555) via cert-manager, but DCERTS ACME returns HTTP 500 on order creation and remains blocked pending vendor resolution. REST API prerequisites (service account, AGS entitlement, firewall rules) must be completed before REST automation is operable.

3.2 Critical Findings

  1. DCERTS ACME Integration Blocked: The ORCHaaS POC observed persistent HTTP 500 errors (AVX_CERT-ACME-008) during ACME order creation. POC artefacts captured a server-side stack trace consistent with a backend failure (java.lang.NullPointerException in AcmeCreateOrderService). Retain an immutable snapshot of the relevant log excerpts and request/response identifiers in the evidence pack. This blocks automated certificate issuance via cert-manager until vendor resolution.

  2. Split-Custody Coordination Gap: No automated mechanism synchronises certificate renewal planning between Cequence and OKAPI teams. This causes expiry date drift (representative examples in Appendix 22) and requires manual coordination.

  3. Secrets Management Fragmentation: EAB keys, RFC 2136 TSIG secrets, and private keys are distributed across AWS Secrets Manager, Sealed Secrets, and (in some workflows) manual file-based storage (e.g., SharePoint) without a single, consistently enforced access-governance model.

  4. Chain Validation Gaps: Legacy endpoint chains may include SHA-1-signed intermediates. These cause compatibility failures with OpenSSL 3.x runtimes (representative example chain + reproducer output in Appendix 22). The trust-chain diagram in 8 is a target-state example (structure and lint objectives), not evidence that all deployed endpoints are already clean; the current prevalence is unknown and must be measured via inventory + automated scans before any statement about risk magnitude or “how widespread” can be made. No automated chain linting is evidenced in the current workflow.

  5. Non-Telstra CA and Custom Domain Controls Absent: No trust anchor allowlist, domain authorisation lifecycle, or tiered governance framework exists for certificates issued by non-Telstra CAs (e.g. Amazon Trust Services via ACM) or for custom domain enablement. Section 7 defines these controls: a phased domain authorisation model (Invariant I7), a CA due diligence checklist with trust profile bindings (Invariant I8), and a tiering model proportional to risk. The ORR gate (Section 4.5.3) provides post-issuance risk assessment via the KUDOS Compare API before any certificate reaches production.

3.3 Highest-Leverage Optimisations

  • Enforce the three-gate model: G1 (CSR lint, pre-issuance), G2 (post-issuance validation + ORR risk assessment via KUDOS Compare), G3 (deployment safety gate, runtime verification)

  • Establish a machine-readable certificate inventory (bootstrap first). Define explicit cutover criteria to an authoritative source of truth once integrity controls are operating

  • Establish pre-approved emergency change templates for key compromise and CA revocation

  • Resolve DCERTS ACME blocker (prerequisite for cert-manager), but do not assume ACME availability in the near-term operating model

  • Deploy the ORR gate (Section 4.5.3) and trust anchor allowlist (Section 7.2) to enable non-Telstra CA onboarding and phased custom domain support

3.4 Effort Reduction: Before and After

Table 2 quantifies the SecEng effort per certificate renewal under the current manual workflow and the target operating model. SecEng coordinates; Cequence Ops (vendor) executes. The operating model eliminates coordination overhead for automated paths and reduces manual path effort through templating and inventory-driven CRP generation.

Activity Current Target How

CSR review and validation 30 min 2 min G1 CSR lint gate (automated ZLint/certlint)

Evidence pack assembly 30 min 0 min Gate outputs auto-captured to evidence ledger

Total per renewal (DCERTS path) 4.3 hrs 0.5 hrs 88% reduction
ACM auto-renewal path 4.3 hrs 0 hrs ACM renews automatically; SecEng monitors via ORR

At 50 renewals per quarter, this reduces SecEng effort from approximately 215 hours to 25 hours (DCERTS path) or near zero (ACM path). The reduction is structural, not incremental: automated gates replace manual verification, and inventory-driven templates replace manual data entry.
