8 DCERTS and ACME Integration

Warning: ACME automation is not currently operable in DCERTS (POC observed HTTP 500 during order creation on 13 May 2025). The near-term operating model therefore assumes manual DCERTS workflow and/or DCERTS REST API patterns.

Detailed cert-manager/ACME artefacts, DCERTS endpoint details, and secrets/network-flow diagrams may exist in Confluence or other internal tooling. For auditability, export these periodically as immutable snapshots (e.g., PDF) and store them with the evidence pack. Where vendor manuals or DCERTS API schemas justify a control or limitation, retain a dated copy alongside the evidence pack.

Readiness gates (before any implementation beyond lab use):

  • DCERTS ACME order creation succeeds consistently in non-prod (no HTTP 5xx) with a documented vendor fix.

  • A single end-to-end non-production pilot has passed (DNS-01, issuance, renewal simulation, revocation/rollback) with evidence captured.

  • Firewall burns / network flows are approved and monitored (to DCERTS ACME + RFC 2136 nameserver).

  • Credential custody is approved (EAB + TSIG + ACME account key stored and rotated via an approved secrets platform).

  • Support model is agreed: on-call ownership, incident runbook, and clear fallback to manual/DCERTS REST API.

Interim automation path (while ACME is blocked): Where renewal automation is needed today, use direct DCERTS REST API integration via a hybrid orchestration model. Power Automate handles email ingestion (native Outlook connector for DCERTS expiry notifications), then hands off via webhook to Rundeck (under evaluation) for orchestration. Rundeck pulls code from GitLab dynamically, providing version control, mandatory security scanning, and auditable execution. This hybrid preserves the same validation gates and evidence pack design, and provides a controlled migration path to cert-manager once DCERTS ACME stabilises. All automation code must reside in GitLab with mandatory pipeline scanning. Manual code paste into cloud UIs is prohibited.

8.1 Transitional Patterns (REST Today \(\rightarrow\) ACME Later)

The DCERTS ACME server is currently unavailable (HTTP 500 during newOrder/order creation). The operating model therefore includes transitional patterns that deliver immediate risk reduction via REST/manual workflows without creating a redesign cliff when ACME becomes available. The design treats ACME, REST, and portal as issuance transport differences behind the same control gates, secret delivery, and evidence/telemetry model.

8.1.1 Controller Handoff Protocol

When a backend changes (e.g., REST/operator \(\rightarrow\) ACME/cert-manager), only one controller may write the live TLS Secret at any time. The handoff protocol: (1) freeze renewals and record current fingerprint, (2) disable the old writer, (3) re-key under the new controller, (4) issue and lint via standard gates, (5) cutover to canary, (6) prove via SNI probe, (7) unfreeze and monitor. Rollback if fingerprint does not match. The three-gate model (G1/G2/G3) applies identically regardless of transport. The detailed step-by-step checklist is maintained in the operational runbooks.

KUDOS Platform (Ruled Out): The KUDOS auto-cert-operator was evaluated but is not applicable to Cequence. The operator writes exclusively to Kubernetes Secrets; Cequence is AWS SaaS without a Kubernetes runtime. The KUDOS team does not support external API consumption. Cequence certificate automation uses direct DCERTS REST API integration (Section 8.1) or the manual DCERTS portal workflow.
