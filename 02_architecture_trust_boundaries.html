6 Architecture Overview

6.1 Split-Custody TLS Model

The Cequence deployment pattern creates a split custody model. Two independent certificate chains terminate TLS for the same customer-facing FQDN. 6 illustrates the complete trust boundary architecture.

The following engineering constraints govern certificate operations across both custody zones.

  • Renewal coordination: Cequence and OKAPI renewals for a given customer-facing FQDN should be planned as a single coordinated change (same release train, same change window). This reduces the probability of one-side-at-a-time certificate incidents.

  • Expiry alignment scope: Aligning issuance dates is not a cryptographic safety property and is neither necessary nor sufficient to prevent expiry-driven outages. The control objective is to ensure both certificates remain valid with adequate buffer and that each side has a tested renewal path. Expiry alignment serves only as a coordination heuristic for shared planning and must not be treated as a hard invariant.

  • Evidence requirement: For each endpoint_fqdn, both teams must record valid_to (NotAfter) for server_edge and server_origin certificates in the inventory and demonstrate (a) independent expiry alerting and (b) an agreed renewal_target that fits the shared change windows.

  • Termination mapping: For each in-scope endpoint_fqdn, maintain an evidence-based TLS termination map for both legs: consumer \(\rightarrow\) defender and defender \(\rightarrow\) upstream (including intermediate terminating components such as CDN/WAF/LB/managed gateways). If the upstream SaaS termination region or site is unknown (or non-AU), treat it as an unresolved constraint for that endpoint until verified and accepted.

  • Independent private keys: Private keys MUST NOT be shared between zones. Each team generates CSRs independently.

  • Chain consistency: Split custody does not permit enforcing identical issuance, but it can enforce chain compatibility. Define an approved CA profile set per environment (e.g., “Globally Trusted” for Internet-facing endpoints) and treat any deviation as an exception requiring (a) a documented consumer trust impact assessment, (b) evidence that major client truststores build to the chain (OpenSSL 3.x, JVM, iOS/Android), and (c) a remediation timeline. Both teams must record issuer_ca (root and intermediate identifiers) in the inventory and pass automated chain validation against a published trust bundle.

  • SNI validation: Defenders must validate SNI against the deployed certificate SAN set to prevent domain mismatch errors.

6.1.1 Split-Custody Automation Model

Each zone owns its private keys and can renew independently. ACME, operators, and Vault-based patterns remain viable, but only as two parallel automation domains (one per custody boundary) with a cross-boundary synchronisation layer that shares metadata, not keys.

Required properties:

  • No cross-boundary key custody: a single Vault cluster may exist, but Cequence and OKAPI key material MUST be isolated by policy and namespace (separate mount paths/tenants, separate auth methods/roles, separate operators), and access MUST be non-transitive across the two operational teams.

  • Separate issuance identities: if ACME is used, each zone uses its own ACME account key and EAB credentials; compromise or revocation in one zone must not impact the other.

  • Shared source of truth for planning (metadata only): both zones publish valid_to, issuer_ca, and serial into the certificate inventory; the inventory becomes the synchronisation surface for renewal planning and incident response (not a place to store private keys).

  • Deterministic renewal handshake: renewals are coordinated via a single change record that references both cert records; each side must (a) renew and deploy independently, and (b) prove post-deploy validation (fingerprint/SNI checks) before the other side proceeds.

  • Failure isolation: automation failure in one zone (ACME outage, operator bug, Vault policy error) must not block the other zone from using the manual DCERTS workflow to renew.

6.1.2 Assertion contract (Cequence to OGW)

OGW cannot observe the client-facing TLS handshake (Section 1). Any session information OGW requires must be explicitly asserted by Cequence via a defined interface. This operating model adopts a minimal assertion contract (Decision A): Cequence asserts only what it can directly verify at its termination point; OGW consumes these assertions without independent handshake verification.

Cequence-to-OGW assertion contract (minimal contract, version 1.0).
Asserted attribute Source (Cequence termination) Evidence requirement
authenticated_principal Client identity verified at Cequence (mTLS client cert subject, API key, or OAuth token) Audit log entry linking request ID to principal extraction
tls_client_verified Boolean: client certificate was presented and validated against the configured trust bundle Probe output confirming mTLS enforcement (known-good/known-bad tests)
request_id Unique identifier generated by Cequence for correlation Logging schema validation (request_id present in all log entries)

Explicit non-claims: OGW must not claim or log as directly observed: TLS version/cipher negotiated with the client, client certificate subject/issuer/serial/SAN (unless asserted by Cequence), full certificate chain contents, or any handshake-level attributes not forwarded by Cequence. Logging such attributes without receiving them via the assertion contract produces false audit evidence and creates compliance risk.

Contract versioning: The assertion contract is versioned (assertion_contract_version in the inventory schema). Changes to asserted attributes require coordinated rollout between Cequence Ops and OKAPI Ops. Version mismatch is a deployment-blocking condition (Invariant I9).

Degraded mode: If the assertion pipeline is unavailable (Cequence cannot forward attributes to OGW), behaviour is fail-closed: OGW rejects traffic that depends on asserted attributes. Fail-open is not permitted because OGW would operate without identity or session context, producing silent security degradation. Cequence logs assertion delivery failures as severity-1 events and triggers the emergency rotation runbook (Section 13.2) scoped to the affected endpoints. Restoration requires proof that assertions resume correctly (synthetic probe with known-good and known-bad inputs) before traffic is re-admitted.

6.2 Certificate Trust Chain

8 illustrates the target-state certificate chain structure for Globally Trusted certificates issued via DCERTS using a modern SHA-256+ intermediate. Not all deployed endpoints currently present this clean chain; in practice, environments contain a mixture of modern and legacy chains (including SHA-1-signed intermediates).

Reconciling “clean DigiCert chain” vs “legacy SHA-1 reality”:

  • Target-state: All customer-facing endpoints present a modern chain (SHA-256+), compatible with OpenSSL 3.x and modern JVMs.

  • Observed failure mode (possible): Some endpoints may still present legacy intermediates (including SHA-1 signatures) which can fail validation in strict client stacks. Prevalence is unknown; no quantified measurement exists.

  • What is missing today: A measured breakdown (% of endpoints) by issuer/intermediate/signature algorithm. Derive this from the certificate inventory and automated endpoint scans before making any risk quantification claim.
