14.2 Engineering verification (Splunk cross-checks)

Operational verification must not rely on a single data plane. Verify incidents and changes using independent log/telemetry sources. This allows claims such as “handshake failures are widespread” to be validated quickly.

Current state: Five production Splunk indexes exist for Cequence (secops-cequence-amp-detection-app, -mitigation-app, -risk-app, -audit-app, and secops-cequenceapi-app for bypass healthcheck alerts). These indexes contain API security events (bot detection, mitigation actions, OWASP risk scores, authentication audit) with fields including host, uri, status_code, transaction_id, client_ip, and fingerprint (bot fingerprint, not certificate fingerprint). No certificate lifecycle telemetry (certificate fingerprint, serial, issuer, expiry, chain composition) is currently present in any Cequence Splunk index. The okapi_logs index exists but returns zero events; cross-team log correlation is not possible through Splunk today.

Cequence defenders do forward the served public certificate (or its fingerprint) in a response header to the upstream platform. This existing data path is the lowest-effort source for building certificate telemetry: extract the certificate metadata at the defender edge, forward it via HEC to a dedicated Splunk index, and correlate against the inventory record_id.

Expectations for Splunk-backed checks:

  • Queries must be scoped by explicit time window and environment.

  • Outputs attached to an evidence pack must include: the query text (or a hash), result count, and representative raw events.

  • Where possible, correlate Splunk events back to the canonical object \(\mathcal{C}\) and the inventory record_id.

  • Until certificate telemetry is available, use the existing detection index (status_code >= 500 with timechart span=15m) as a proxy for TLS-related service degradation.

15 Non-Kubernetes Estate Strategies

The implementation artefacts primarily target Kubernetes (cert-manager, operators, Secrets). For non-Kubernetes estates (VM-based workloads, legacy middleware, appliance-based terminators), the same control gates and evidence requirements apply. Only the delivery mechanism differs.

Control-layer equivalence:

  • The three-gate model (G1: CSR lint, G2: post-issuance validation + ORR, G3: deployment safety gate) applies identically.

  • Inventory schema, evidence pack structure, and telemetry hooks remain the same.

  • Split-custody boundaries and PKI/DCERTS approval routing are unchanged.

Delivery-layer differences:

  • Secret delivery: Replace Kubernetes Secrets with HashiCorp Vault (AppRole/agent), AWS Secrets Manager, or Azure Key Vault. Ensure the same tls.crt/tls.key naming convention for portability.

  • Renewal trigger: Replace cert-manager with scheduled jobs (cron, systemd timers, or orchestrator-based) that invoke the DCERTS REST API directly or manual DCERTS workflow.

  • Deployment: Replace Kubernetes rolling updates with configuration management (Ansible, Chef, Puppet) or load-balancer drain/swap patterns.

  • Validation probes: Replace Kubernetes readiness probes with external synthetic monitors (Datadog, Prometheus blackbox exporter) targeting the same SNI/fingerprint checks.

Any non-Kubernetes implementation must produce the same evidence artefacts (Table 19). It must also pass the same deployment safety gate (Table 18) before a change can close.
