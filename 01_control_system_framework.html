4 Operating Model as a Control System

The operating model is structured as a control system. The central problem is keeping runtime TLS behaviour within acceptable bounds (availability, compatibility, custody, auditability). Split ownership, vendor gates, and changing client truststores all push against this.

4.1 Canonical Identifiers and Correlation Keys

Canonical certificate object: \[\begin{equation*} \mathcal{C} = (\texttt{endpoint\_fqdn}, \texttt{environment}, \texttt{certificate\_role}) \end{equation*}\]

All automation, tickets, evidence, and telemetry correlate to \(\mathcal{C}\) and the inventory record_id. A signal that cannot be joined to \(\mathcal{C}\) is operationally useless under incident pressure.

4.2 Control Invariants

Hard invariants: the system properties that prevent “quiet” certificate failures.
Invariant Statement (enforceable)
I1: Runtime truth “Deployed” means the served leaf fingerprint is observed via SNI probe; portal/inventory values are advisory until proven by runtime evidence
I2: No split-brain For each \(\mathcal{C}\), exactly one controller is authoritative to mutate the live TLS material at any time (manual, operator/REST, or cert-manager)
I3: Gate closure No change record can close without the deployment safety-gate artefacts (Table 18); restart is exception-managed
I4: Custody boundary Private keys never cross custody domains; inventory and workflows handle metadata only
I5: Approval boundary PKI/DCERTS remains R/A for issuance approval across all transports (portal/REST/ACME); automation may route and track but must not auto-approve
I6: Correlation or it did not happen Every alert, ticket, evidence pack, and probe output must carry \(\mathcal{C}\) and record_id (or be rejected/triaged as “untracked”)
I7: Domain authorisation No SAN/FQDN addition or change without authorisation evidence attached to \(\mathcal{C}\); authorisation artefact must reference ownership proof and validation method
I8: Trust profile explicitness Every \(\mathcal{C}\) declares server_trust_profile_id and, if mTLS is enabled, client_trust_bundle_id; defaults must be explicit (no implicit “any CA”)
I9: Assertion contract versioned OGW-facing asserted attributes are versioned (assertion_contract_version) and evidence-backed; no attribute may be asserted without a defined verification procedure

4.4 Controller authority model (anti-race-condition rule)

Controller conflicts are a dominant root cause for certificate incidents in automated estates (key overwrite, competing renewals, non-deterministic rollback). The authority model makes this explicit:

  • Authority is per \(\mathcal{C}\): one authoritative writer for the live TLS material (manual process, operator/REST, or cert-manager).

  • Authority is switched only by protocol: cutovers must follow the handoff checklist in Section 8.1.1 (freeze, disable old writer, re-issue under new controller, prove by fingerprint).

  • Dual-run is permitted only on disjoint objects: multiple controllers may exist in the estate, but must operate on non-overlapping \(\mathcal{C}\) sets.

4.5 Execution Architecture

The patterns compress into four composable machines. Each machine can be partially or fully automated while preserving the invariants in Table 4.

4.5.1 M1: Triage Machine (alerts \(\rightarrow\) ownership \(\rightarrow\) classification)

Inputs: DCERTS notifications; synthetic probe alerts; incident tickets; KUDOS Compare pre-renewal risk baseline (comparison of expiring certificate against inventory record).

Outputs: a tracked object \(\mathcal{C}\) with record_id; owner_team; SLA classification (Table 21); pre-renewal risk assessment attached to record_id.

Gates:

  • If owner_team cannot be assigned, the system is in a “blind” state: escalation is mandatory (no silent backlog).

  • If record_id does not exist, the incident is operationally uncorrelatable: create it first (or treat as a defect in inventory integrity).

4.5.2 M2: Issuance Machine (CSR lint \(\rightarrow\) approval \(\rightarrow\) issued artefact normalisation)

Inputs: CSR + metadata for \(\mathcal{C}\) (SAN set, CA profile, change record).

Known constraint: Cequence support may deliver CSRs as compressed archives with nested zip structures. The CSR lint gate input normalisation step must handle archive extraction before validation. Format standardisation with the vendor is a coordination item.

Outputs: issued leaf+chain artefacts; DCERTS request/transaction identifiers; captured NotBefore/NotAfter, serial, issuer chain identifiers.

Gates:

  • G1: CSR lint gate must pass before submission (Table 17).

  • Approval boundary is strict: PKI/DCERTS approval remains mandatory (Invariant I5).

  • G2: Post-issuance validation + ORR risk assessment must pass before deployment (Section 11.2, Section 4.5.3).

4.5.3 ORR Gate: Post-Issuance Risk Assessment (blocking)

After issuance, the G2 gate validates the certificate and assesses deployment risk. No certificate reaches production without G2 clearance. For renewals, KUDOS Compare provides integrated chain/algorithm validation alongside risk scoring. For onboardings, standalone cert lint (Section 11.2) runs first. “PKI Ops” refers to the PKI/DCERTS operations team (the same entity referenced in Invariant I5).

Mechanism: The KUDOS Compare API (/api/compare) accepts old and new certificates (PEM-encoded) and returns a risk score (integer) plus a findings array (field-level differences with severity tags such as INFO). The compare output feeds the ORR autofill URL via query parameters (serial number, chain status, algorithm status). Rundeck then submits the populated ORR form to the Confluence-hosted ORR workflow. The certificate comparison and the ORR submission are distinct steps: the former produces the risk assessment, the latter acts on it.

Risk branching:

  • Risk score \(< 60\) (low risk): ORR clears automatically. No ORR Jira ticket created (per SHO-4039). Proceeds to M3.

  • Risk score \(\geq 60\) (high risk): PKI Ops receives the assessment, creates an ORR Jira ticket ([orr_jira_created]), and reviews the deployment. PKI Ops approval is required before transition to M3.

Orchestration note: Two distinct Jira tickets may exist for a single \(\mathcal{C}\): the CERTLIFECYCLE ticket (created by Rundeck at triage, always present) and the ORR ticket (created only for high-risk assessments). Both carry record_id for correlation.

Constraint: The ORR gate is blocking; deployment cannot proceed without a risk score. Timeout and SLA handling are implementation-level concerns documented in the integration specification.

Integrated validation model (renewals): For renewals, the ORR step subsumes standalone cert lint. KUDOS Compare validates chain status and algorithm compliance for both old and new certificates as part of its comparison. An AIA/CRL reachability probe runs in parallel (a network check that no comparison API can perform). Together, these satisfy the G2 validation specification (Section 11.2) without a separate lint gate.

Renewal flow: extract served cert from defender traffic headers (oldCert) \(\rightarrow\) retrieve newly issued cert from DCERTS API (newCert) \(\rightarrow\) KUDOS Compare (validates both, computes risk score) + parallel AIA/CRL probe \(\rightarrow\) ORR submission.

Onboarding flow: no existing cert for comparison. Standalone cert lint runs first (Section 11.2), then KUDOS Compare runs against a reference baseline certificate.

Preconditions (must be verified before adopting integrated model):

  • Traffic header contains the full leaf certificate (not truncated) and is consistently present across all defender configurations.

  • KUDOS Compare coverage is confirmed against every check in Section 11.2 (chain completeness, algorithm compliance, intermediate expiry). Any gap means standalone lint must remain.

  • AIA/CRL reachability is checked as a parallel probe, not delegated to KUDOS Compare.

Fallback: Until KUDOS Compare coverage is empirically verified, standalone cert lint (Section 11.2) runs as a separate step before KUDOS Compare. The three-gate model (G1: CSR lint, G2: post-issuance validation + ORR, G3: deployment safety gate) applies regardless of whether cert lint is standalone or integrated.

4.5.4 M3: Deployment/Verification Machine (canary \(\rightarrow\) fingerprint rollout \(\rightarrow\) SLO validation)

Inputs: certificate package (leaf+chain); rollback package; deployment targets; G2 clearance (low risk: automatic; high risk: PKI Ops approval via ORR Jira).

Outputs: new served fingerprint observed; handshake success/latency time series; DCERTS portal deployment confirmation (manual portal update recording deployment completion).

Gates: G3 deployment safety gate (Table 18) and evidence mapping (Table 19). DCERTS deployment confirmation is orthogonal to G2; it is a bookkeeping step performed after verification passes.

4.5.5 M4: Emergency Machine (freeze \(\rightarrow\) expedited issuance \(\rightarrow\) minimum viable proof)

Inputs: incident declaration with scoped fingerprints/issuers/SNI.

Outputs: freeze asserted; emergency issuance request; minimum evidence artefacts to re-establish safe runtime.

Gates: emergency patterns E1–E3 in Section 13 must remain executable under time pressure; the system is designed so evidence is produced by default rather than “remembered later”.

4.6 Append-only Evidence Ledger

A certificate operating model fails when teams can say a control happened without leaving a reliable trail. Evidence is therefore treated as a first-class data product.

Ledger concept: an append-only sequence of entries capturing the runtime truth and the verification work that produced it. Each entry is immutable and hash-addressable.

Implementation direction: The project owner mandated an immutable ledger. The leading candidate is a git-based audit log: every certificate lifecycle event committed as an append-only JSON record to a dedicated GitLab repository. This approach has zero licensing cost, is inherently immutable, and aligns with the Rundeck + GitLab architecture. Alternative candidates (PostgreSQL with append-only triggers, Azure Event Hubs, AWS QLDB) remain under evaluation. The ledger is separate from the SharePoint Lists operational inventory; inventory tracks live state, the ledger provides tamper-evident history.

Evidence ledger: minimum fields for tamper-evident operational truth.
Field Type Purpose
ledger_ts DateTime Ordering and incident correlation
record_id UUID Stable join key to inventory and tickets
canonical_object Tuple \(\mathcal{C}\) = (fqdn, environment, certificate_role)
actor_id String Human or controller identity
probe_cmd_hash String Hash of command/script used (reproducibility)
probe_out_hash String Hash of stdout/stderr (tamper evidence)
served_fingerprint String Runtime truth for “what is deployed”
notAfter_observed DateTime Runtime NotAfter via probe (drift detection)
notes String Optional reason codes / exception reference

Pragmatic implementation note: the ledger can be implemented in multiple backends. The requirement is the semantics (append-only, immutable history, and hashes), not a particular product.

4.7 ACME Migration Safety Requirements

ACME adoption changes the control plane. This model treats it as a safety-critical migration because it changes (a) who can write TLS material, (b) how keys are generated, and (c) how renewals are triggered.

Safety property: migrating issuance backends MUST NOT weaken the invariants in Table 4. Therefore, any transition (manual \(\rightarrow\) REST/operator \(\rightarrow\) ACME/cert-manager) is gated by:

  • Controller handoff protocol execution (Section 8.1.1).

  • Runtime proof: served fingerprint roll-forward after cutover.

  • A rollback path that returns authority to the previous controller without ambiguity.

4.8 Implementation realism

The control-system framing is compatible with current constraints: no Zendesk API automation, ACME blocked, split custody, and mandatory PKI/DCERTS approvals. It yields immediate risk reduction by making failure states visible (runtime truth, no split-brain, gated closure) even when issuance remains partially manual.

The ORR gate (Section 4.5.3) operates within the existing manual workflow; the KUDOS Compare API and ORR form are already operational tools. The Rundeck + GitLab hybrid (under evaluation) adds auditability and version control without new infrastructure. The git-based ledger candidate uses the same GitLab instance. Each integration adds control surface without requiring the blocked ACME path.

5.2 Australian Telecommunications Compliance (Non-normative prompts)

This subsection is not a compliance baseline. It contains non-normative prompts (questions to validate) that surface where certificate operations could be relevant to telecommunications regulation. It must not be used as a requirements source, audit baseline, or control catalogue until validated by Legal/Compliance and evidenced for the specific service/environment.

Potential intersections to validate (not requirements):

  • Security of Critical Infrastructure Act 2018 (SOCI): Confirm whether the relevant services/assets are in scope, and whether certificate/key operations are implicated in any applicable obligations.

  • ACMA carrier licence / reporting conditions: Confirm whether certificate-driven availability incidents change reporting thresholds, notification timelines, or incident classification requirements.

  • Telecommunications (Interception and Access) Act 1979: Confirm whether private key handling, privileged access, and logging/audit controls have any specific expectations or constraints under TIA-related obligations.

Any downstream “MUST/SHALL” statements must be sourced from validated Legal/Compliance advice and the applicable regulatory text, and recorded with service-specific evidence.
