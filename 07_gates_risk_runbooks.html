11 Mandatory Control Gates

Three gates enforce certificate quality at distinct lifecycle positions. G1 (pre-issuance) prevents bad requests from reaching DCERTS. G2 (post-issuance) validates the issued certificate and assesses deployment risk via the ORR workflow. G3 (deployment) verifies runtime behaviour. Each gate exists at a different trust boundary; none is redundant with the others.

11.1 G1: CSR Lint Gate (Pre-Issuance)

CSR lint gate validation checks.
Check Validation Rule Failure Action
SAN allowlist CN and SANs match approved domain list Reject with domain owner escalation
Key algorithm RSA 2048+ or ECDSA P-256+ Reject with key regeneration required
Subject constraints O, OU, C, ST, L match organisation policy Reject with subject correction
EKU correctness serverAuth for server certs Reject with EKU correction
Signature algorithm SHA-256 or stronger Reject with CSR regeneration

11.2 G2: Post-Issuance Validation Specification

The following checks must pass after a certificate is issued and before it may proceed to deployment. They form the coverage specification for the ORR-integrated validation step (Section 4.5.3).

For renewals: the KUDOS Compare API (Section 4.5.3) validates chain status and algorithm compliance as part of its old-vs-new comparison. An AIA/CRL reachability probe runs in parallel. Together, these satisfy G2 without a separate standalone lint step.

For onboardings (no existing cert): standalone cert lint runs first (no old cert for comparison), then KUDOS Compare runs against a reference baseline. Both must pass.

Objective (per 8): chain validation MUST be explicit and repeatable. “Certificate was issued” does not satisfy this requirement.

Minimum validation checks (MUST pass for onboarding/renewal):

  • Complete chain builds: leaf \(\rightarrow\) intermediate(s) \(\rightarrow\) trusted root (validated against the intended trust bundle).

  • No SHA-1 signatures anywhere in the served chain: leaf and intermediate(s). (Root signature is typically irrelevant for path building, but if included in the served bundle it should also be non-SHA-1.)

  • No expired intermediates: any intermediate in the served chain MUST have NotAfter in the future at deployment time.

  • AIA/CRL endpoints reachable: URLs advertised in AIA/CRL Distribution Points MUST be reachable from the relevant runtime networks, and this reachability MUST be monitored continuously (not only at change time).

# Inputs
LEAF=server.crt
CHAIN=chain.pem              # intermediate(s) (and optionally root) as a PEM bundle
TRUST_BUNDLE=ca-bundle.pem   # trust anchors used by your clients

# 1) Chain builds (fails if chain is incomplete or does not link to trust)
openssl verify -CAfile "$TRUST_BUNDLE" -untrusted "$CHAIN" "$LEAF"

# 2) Split PEM bundle into individual certs and lint each cert for:
#    - SHA-1 signature
#    - Expiry (NotAfter)
TMPDIR=$(mktemp -d)
cat "$LEAF" "$CHAIN" > "$TMPDIR/fullchain.pem"
awk 'BEGIN{c=0}/-----BEGIN CERTIFICATE-----/{c++}{print > ("'$TMPDIR'/cert-" c ".pem")}' "$TMPDIR/fullchain.pem"

for f in "$TMPDIR"/cert-*.pem; do
  echo "=== $f ==="
  openssl x509 -in "$f" -noout -subject -issuer -dates

  SIG=$(openssl x509 -in "$f" -noout -text | grep -m1 "Signature Algorithm")
  echo "$SIG"
  echo "$SIG" | grep -qi "sha1" && { echo "FAIL: SHA-1 signature detected"; exit 1; }

done

# 3) AIA/CRL endpoint reachability (best-effort reachability probe)
#    Note: enforce in monitoring too; do not rely on change-time checks only.
for url in $(openssl x509 -in "$LEAF" -noout -text \
  | grep -Eo '(http|https)://[^ ]+' \
  | sed 's/[),]$//' \
  | sort -u); do
  echo "Probing: $url"
  curl -fsS --max-time 10 "$url" >/dev/null || { echo "FAIL: unreachable AIA/CRL URL: $url"; exit 1; }
done

rm -rf "$TMPDIR"

11.3 G3: Deployment Safety Gate (Runtime Verification)

Certificate changes are high-risk deployment events. Failures are immediately user-visible (handshake errors) and hard to remediate if recovery requires a service restart or re-issuance. Rotation MUST be deployable as a mechanised change with explicit pass/fail criteria, measurable safety margins, and defined rollback triggers. No service restart is expected for certificate rotation. If a restart is required, it MUST be treated as an exception with time-bound risk acceptance, compensating controls, and a tested downtime plan.

Gate inputs (what must exist before attempting deployment):

  • New leaf certificate + full chain package (leaf + intermediate(s)); previous package retained for rollback.

  • Inventory record for the endpoint including endpoint_fqdn, certificate_role, issuer_ca, valid_to, and rollback_ref.

  • A pre-change baseline window for handshake success rate and handshake latency (minimum: 15 minutes).

Deployment safety gate checks (enforced, measurable).
Check How to measure (mechanistic) Pass condition Failure action
Fingerprint rollout Synthetic TLS handshake (SNI) captures served leaf fingerprint pre/post change New fingerprint observed within 5 minutes of deployment start Block/rollback: deployment did not take effect or is serving stale cert
No restart required Observe process start time / container ID / pod restart count for the terminating component 0 restarts attributable to the cert change Block: raise platform defect unless an approved exception exists
Handshake success Synthetic probe success rate for \(\geq 10\) minutes during canary Success rate \(\geq 99.9\%\) and no sustained failures Rollback: revert to previous certificate package
Handshake latency Synthetic probe p95 handshake time compared to baseline window p95 increase \(\leq 20\%\) vs baseline Rollback: revert and investigate chain/OCSP/AIA causes
Chain lint Daily/inline chain build validation + signature algorithm check Chain builds; no SHA-1; intermediates not expired Block: repackage/reissue chain
SNI/SAN match Synthetic handshake uses target SNI; validate SAN contains SNI SNI present in SAN allowlist Block: fix cert request/config
mTLS behaviour (if enabled) Two probes: (a) known-good client cert, (b) known-bad cert (a) succeeds, (b) fails as expected Block: fix trust bundle/EKU or client auth policy
Rollback readiness Exercise rollback in non-prod or validate artefacts + automation path Rollback is executable within 5 minutes Block: do not enter change window

If the “No restart required” artefact cannot be produced, the deployment is blocked unless an approved exception exists (documented downtime plan, compensating controls, and post-change verification).

Implementation hooks (where this gate lives):

  • Pre-deploy (CI): run CSR/cert lint (Sections 11.1/11.2) and chain validation against the intended trust bundle.

  • Deploy (CD): apply the secret/config change as a canary to a single defender/origin pool first.

  • Post-deploy verification (automated): run the synthetic handshake probes (Appendix 20) against the canary and then against full fleet; record fingerprints and success/latency metrics as evidence.

  • Continuous monitoring: keep the same probes running as SLO telemetry (Section 8.3) so the gate and monitoring use the same signals.

Minimum evidence artefacts (attach to the change record):

  • Pre-change and post-change served leaf fingerprint (per endpoint_fqdn), plus NotAfter date.

  • Canary window metrics: handshake success rate and p95 handshake latency, with baseline window shown.

  • Confirmation of zero restarts (pod restart count / process start time) for the terminating component.

  • Chain validation output (openssl verify plus signature algorithm check) for the deployed package.

Rollback triggers (must be configured before change start):

  • Immediate rollback if (a) fingerprint does not change within 5 minutes, (b) handshake success \(<\) 99.9% for \(\geq 2\) consecutive minutes, or (c) p95 handshake latency increases \(>\) 20% for \(\geq 5\) minutes.

  • Rollback action: redeploy the previous certificate package (leaf + full chain) and revert any associated config change; validate by fingerprint + synthetic SNI checks.

  • Rollback RTO target: restore baseline handshake success rate within the change window (target: \(\leq 5\) minutes per endpoint).

12 Failure Mode Analysis

12.1 Risk Matrix

Likelihood ratings in 20 are not subjective priors unless labelled as such. Each value must derive from a defined measurement window and dataset. Values without measured data must be marked Assumed prior (unmeasured).

Measurement approach:

  • Observation window: trailing 12 months (or the maximum available period if less), reported with explicit start/end dates.

  • Datasets:

    • Incident tickets for Cequence/OKAPI tagged as certificate/TLS related (expiry, chain, truststore, handshake, mTLS).

    • DCERTS availability events (vendor/outage notifications, internal incident records) and/or API error-rate telemetry (e.g., 5xx rate for /cert/create, /cert/renew, ACME order creation).

    • Certificate inventory + automated endpoint scan results (e.g., % of endpoints within 30/14/7 days to expiry; % presenting legacy intermediates; % failing chain validation under OpenSSL 3.x).

  • Mapping to ratings: convert observed frequency into coarse bins (e.g., High = monthly+; Medium = quarterly; Low = annual/rare), and record the computed rate in the evidence pack.

13 Operational Runbooks

13.1 Standard Renewal Runbook

  1. Detection (T-90 days): Receive DCERTS expiry notification; create an internal Jira issue (system-of-record) and link it to the inventory record_id

  2. Planning (T-85 days): Coordinate with OKAPI Ops to agree renewal targets and a shared change window (issuance/expiry alignment is optional); confirm independent monitoring and rollback paths. For mTLS endpoints (certificate_role=client_mtls), confirm that client-side certificate holders are notified and schedule coordinated rotation so that both server and client certificates are updated within the same window. MASSL coordination SLO: client-side certificate holders must confirm readiness by T-30 days; if no confirmation by T-30, escalate to L4 on-call. MASSL renewal failures are the most common coordination failure mode

  3. Vendor engagement (T-75 days): If Cequence action is required, raise a Zendesk ticket / engagement-form request manually and record the vendor ticket ID back in Jira (assume no Zendesk API)

  4. CSR Review (T-70 days): Validate CSR correctness via DCERTS portal; store in SharePoint CSR folder

  5. Issuance (T-60 days): Submit certificate request via DCERTS; await PKI team approval

  6. Staging (T-45 days): Provide the issued public certificate package (leaf + chain only; never private keys) to Cequence support for staging validation

  7. Change Execution (T-14 days): Execute change during scheduled window; perform PVT with API consumers

  8. Closure (T+1 day): Update DCERTS metadata; attach the evidence pack; refresh the certificate inventory; close tickets

13.2 Emergency Rotation Runbook

  1. Declare incident: Scope affected endpoints by SNI, certificate fingerprint, and issuer

  2. Freeze changes: Halt non-essential changes; start evidence capture

  3. Generate new key: Create new key pair under approved custody model

  4. Request issuance: Submit DCERTS request under approved CA profile (use emergency change template)

  5. Deploy with validation: Stage and deploy with synthetic tests and monitored rollout

  6. Revoke compromised: If required, revoke old certificates and update truststores

  7. Post-incident review: Complete PIR; update inventory and evidence pack

13.2.1 Dedicated emergency workflow patterns (required)

Emergency rotation is not a faster standard renewal. It requires dedicated workflow patterns. Teams must be able to (a) stop unsafe concurrent changes, (b) pre-populate approvals and tickets, and (c) generate a deterministic evidence pack under time pressure.

  1. E1. Freeze/hold signal (change stop-the-world): Define an explicit freeze signal that can be asserted and observed across teams and toolchains (e.g., an “Emergency TLS Freeze” Jira label/state + a Change Advisory broadcast). Gate all certificate deployments (edge and origin) on the freeze state: only emergency-approved changes may proceed until the freeze is lifted.

  2. E2. Emergency template pre-population (one-click ticketing): Maintain pre-approved emergency change templates and Jira issue templates. These must auto-populate the minimum fields required for certificate incidents: endpoint_fqdn, current served fingerprint, issuer chain identifiers, suspected failure mode (expiry/compromise/mis-issuance/chain), owner_team, and the required validation artefacts from Table 18. The template MUST include a mandatory linkage to the inventory record_id so alerts, tickets, and evidence remain correlated.

  3. E3. Evidence-pack automation (timestamped artefacts by default): For emergency rotations, require a minimal, deterministic evidence pack that is produced automatically where possible: (a) pre-change and post-change served fingerprint + NotAfter, (b) chain lint output (openssl verify + SHA-1 check), (c) canary handshake success/latency window, and (d) “no restarts” proof where applicable. The evidence pack pointer is then attached to the Jira issue and referenced in the DCERTS request/change record.

SLO: Target 4 hours to stable state for critical incidents.
